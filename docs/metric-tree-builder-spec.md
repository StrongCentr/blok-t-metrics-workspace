# Конструктор метрик — спецификация прототипа

**Версия документа:** 1.1  
**Дата:** 13.07.2026  
**Артефакт:** `metric-tree-builder.html` (один HTML-файл, без сборки)  
**Публикация:** https://strongcentr.github.io/blok-t-metrics-workspace/metric-tree-builder.html  

> **Актуализация:** этот документ — единый источник спецификации конструктора метрик. При изменениях `metric-tree-builder.html` или связанной логики обновляйте этот файл в той же сессии/PR. Отдельный spec для `Курсор.html` пока не ведётся.

Документ описывает **текущую логику прототипа** и предназначен для:
- передачи бизнес-требований команде разработки;
- формирования технических требований (TRD / backlog);
- согласования методологии дерева метрик до промышленной реализации.

---

## 1. Назначение и ценность

### 1.1. Проблема

Продуктовые и бизнес-команды часто:
- не имеют единого дерева метрик под конкретный продукт;
- путают vanity-метрики с метриками результата;
- не связывают финансовую цель с клиентскими, продуктовыми и технологическими рычагами;
- не фиксируют страхующие метрики (guardrails).

### 1.2. Решение прототипа

Инструмент помогает **неспециалисту** за 1–2 минуты получить **черновик дерева метрик** по текстовому описанию продукта, доработать его вручную, проверить качество и экспортировать результат.

### 1.3. Целевая аудитория

| Роль | Сценарий |
|------|----------|
| Владелец продукта / бизнес-лидер | Первичная структура метрик под продукт или инициативу |
| Продакт-менеджер | Уточнение связей, формул, владельцев |
| Аналитик / BI | Валидация логики перед внедрением в дашборды |
| Методолог / консультант | Демонстрация подхода «дерево метрик» |

### 1.4. Что прототип **не** делает

- Не подключается к реальным данным и DWH.
- Не считает значения метрик.
- Не хранит деревья на сервере (только `localStorage` браузера).
- В публичной версии **не генерирует** дерево под произвольный продукт без внешнего AI (см. §6).

## 2. Область продукта (scope)

### 2.1. Включено в прототип

| Модуль | Описание |
|--------|----------|
| **Конструктор метрик** | Ввод описания → построение дерева → редактирование → инсайты |
| **Библиотека метрик** | Справочник готовых карточек с формулами и антипаттернами |
| **Пульс метрик** | Саммари и выводы для бизнес-лидера: топ-3 + лента с фильтрами (`metrics-news.json`) |
| **Экспорт** | JSON, SVG, PNG, HTML |
| **Локальное сохранение** | Восстановление сессии в том же браузере |

### 2.2. Исключено (намеренно)

- Авторизация и многопользовательский доступ.
- Версионирование деревьев.
- Интеграция с корпоративным каталогом метрик / BI.
- Серверный proxy для LLM (запланировано отдельно).

### 2.3. Связанные артефакты (вне этого файла)

- `Курсор.html` — отдельный прототип «Рабочее место лидера» (dashboard, scorecard).
- Деплой обоих инструментов: `index.html` (хаб) на GitHub Pages.

## 3. Методология дерева метрик

### 3.1. Пять слоёв (сверху вниз)

| Уровень | Название в UI | Смысл |
|---------|---------------|-------|
| L1 | Цель | Главная метрика / финансы продукта |
| L2 | Клиенты | Драйверы клиентской базы |
| L3 | Активн. | Активность, использование, продуктовые входные |
| L4 | Рычаги | То, чем команда может управлять (цена, каналы, качество данных) |
| L5 | Тех | Технологические ограничения и enablers |

Слой задаёт **цвет бейджа** узла и семантику, но **не фиксирует** вертикальную позицию на холсте: раскладка строится по связям (§7.3).

### 3.2. Типы метрик (role)

Внутренние ключи → подписи в UI:

| Ключ | Подпись для пользователя |
|------|----------------------------|
| `output` | результат |
| `driver` | часть расчёта |
| `input` | рычаг команды |
| `guardrail` | ограничение / риск |
| `leading` | ранний сигнал |

### 3.3. Типы связей

| relation | Подпись | Смысл |
|----------|---------|-------|
| `direct` | прямая (часть расчёта) | Нижняя метрика входит в формулу / арифметику верхней |
| `indirect` | косвенная (влияние) | Влияет, но не является прямой компонентой расчёта (напр. NPS → отток) |

**Визуализация:** прямые — сплошная линия; косвенные — пунктир (по умолчанию скрыты, включаются кнопкой «Показать все влияния»).

### 3.4. Статус метрики «уже считаем»

- Поле `isCurrent: true` — узел с **сплошной** рамкой.
- `isCurrent: false` — **пунктирная** рамка («не считаете» / предложено деревом).
- Определяется: явным списком метрик во вводе + эвристиками при генерации.

### 3.5. Принципы качества дерева (бизнес-правила)

1. На вершине — **одна** главная метрика (apex без исходящих связей на L1).
2. Финансовые и клиентские драйверы должны **раскладываться** нижними метриками.
3. Обязательны **страхующие** метрики (отток, NPS, доступность, риски, экономика).
4. Результаты и ограничения не должны быть **прямой** частью чужой формулы без осмысления.
5. У каждой «листовой» метрики — владелец, формула, источник (рекомендуется).
6. Vanity-метрики (LOC, PR count, регистрации без активации и т.п.) — сигнал внимания.

## 4. Пользовательские сценарии (user stories)

### US-01. Построить дерево с нуля

**Как** владелец продукта  
**Хочу** описать продукт текстом и получить черновик дерева  
**Чтобы** быстро увидеть пробелы в измерениях  

**Шаги:**
1. Ввести описание в `f_desc`.
2. (Опционально) указать уже считаемые метрики в `f_metrics`.
3. Нажать «Построить дерево метрик».
4. Страница прокручивается к блоку дерева (`#treeCard`).

**Критерии приёмки:**
- Дерево содержит 18–30 узлов (в AI-режиме) или фиксированный набор архетипа (локально).
- Появляется блок «Инсайты и зоны внимания».

### US-02. Использовать пример

**Шаги:** кнопка «Мобильный подписочный сервис» или «SberVision» → заполнение полей → построение.

### US-03. Редактировать метрику

**Шаги:** клик по узлу → drawer с карточкой → изменение полей → «Сохранить».

**Поля карточки:** название, слой, тип, категория, «уже считаем», what, formula, source, freq, owner, thresholds, antipatterns, sources, roi.

### US-04. Добавить / удалить метрику и связь

- «+ Метрика» — новый узел (`userAdded: true`).
- «+ Связь» — режим link: источник → цель, выбор direct/indirect.
- Удаление узла каскадно удаляет связи.

### US-05. Проверить размещение метрики

В карточке: «Разбор метрики» — rule-based анализ; «Что добавить ниже?» — подсказки из библиотеки; «Спросить внешний AI» — при наличии API-ключа.

### US-06. Обсудить новую метрику (консультант)

Блок «Обсудить метрику»: проверка дубля, совпадения с библиотекой, vanity, рекомендуемый слой. При API-ключе — дополнение от LLM.

### US-07. Библиотека → дерево

Вкладка «Библиотека метрик» → поиск → карточка → «Добавить в дерево».

### US-08. Экспорт

Меню «Экспорт»: JSON (полная модель), SVG/PNG/HTML (визуализация).

### US-09. Восстановление сессии

При повторном открытии в том же браузере — восстановление из `localStorage` (`metric-tree-builder-v2`).

### US-10. Пульс метрик (новости)

**Аудитория:** бизнес-лидер.

**Шаги:** вкладка «Пульс метрик» → редакционный топ-3 «Главное за неделю» (саммари + выводы) → лента с краткими саммари и фильтром по темам → ссылка «Оригинал» на источник.

**Данные:** `metrics-news.json` (поля `summary`, `topics`, `takeaways` для featured) или встроенный fallback при `file://`.

**Фильтры ленты:** Все, Scorecard, Финтех, Продукт, Инженерия, AI.

**Критерии:** дата обновления видна; саммари читается без перехода на сайт; оригинал открывается в новой вкладке; экспорт скрыт на этой вкладке.

## 6. Режимы генерации дерева

### 6.1. Локальный режим (по умолчанию)

**UI:** «Локально (без внешнего AI)» (`f_mode = demo`).

**Алгоритм:**
1. `collectInput()` — описание + список метрик.
2. `pickArchetype(text)` — подсчёт совпадений с regex-сигналами:
   - `ai_knowledge` — знания, инсайты, LLM, граф, discovery, гипотезы…
   - `pnl` — выручка, подписка, churn, CAC, клиенты…
   - при равенстве → `pnl`.
3. Выбор шаблона `ARCHETYPES[archetype]`:
   - **pnl** — ~24 узла (PnL-подписочная модель),
   - **ai_knowledge** — knowledge/AI-продукт (SberVision-подобный).
4. Узлы копируются из `LIBRARY` по `nodeIds`.
5. `detectCurrent()` — подсветка `isCurrent` по явному списку и KEYWORDS.
6. Связи берутся из предопределённого списка `edges` архетипа.

**Ограничение:** произвольный новый продукт **не** получает уникальное дерево — только один из двух архетипов.

### 6.2. Режим внешнего AI

**UI:** «Через внешний AI» + endpoint, model, API key.

**Алгоритм:**
1. `callAi(input)` — POST OpenAI-совместимый chat/completions.
2. System prompt `AI_SYSTEM` — методология, 5 слоёв, финтех-сегменты, JSON-схема ответа.
3. `response_format: { type: "json_object" }`.
4. `normalizeTree()` — валидация, нормализация id/level/relation, merge isCurrent.

**Ограничение для GitHub Pages:** ключ в браузере небезопасен; для production нужен proxy (GigaChat / корп. LLM).

## 7. Техническая архитектура прототипа

### 7.1. Стек

| Компонент | Реализация |
|-----------|------------|
| UI | HTML5 + CSS (custom properties, dark theme) |
| Логика | Vanilla JavaScript (ES6+), один `<script>` |
| Граф | SVG + ручная раскладка |
| Хранение | `localStorage` |
| Сборка | Нет |
| Backend | Нет |

### 7.2. Структура экранов

```
header: tabs [Конструктор | Библиотека] + Экспорт
view-build:
  inputCard (hero, onboarding, samples, textarea, metrics, generate, AI settings)
  treeCard (toolbar, guide, canvas/svg, empty state)
  insights
  consult
view-library:
  search + libContent (категории → карточки)
drawer + overlay (карточка метрики / редактор)
```

### 7.3. Модель данных дерева

```typescript
interface MetricTree {
  product: string;      // название продукта
  summary: string;      // краткое резюме (от AI)
  nodes: MetricNode[];
  edges: MetricEdge[];
}

interface MetricNode {
  id: string;
  name: string;
  level: 1..5;
  role: "output" | "driver" | "input" | "guardrail" | "leading";
  category: string;     // id из CATEGORIES
  what, formula, source, freq, owner, thresholds, antipatterns, sources, roi: string;
  isCurrent?: boolean;  // уже считаем
  userAdded?: boolean;  // добавлено пользователем / из библиотеки
}

interface MetricEdge {
  from: string;         // id источника (ниже по влиянию)
  to: string;           // id цели (выше)
  relation: "direct" | "indirect";
}
```

**Направление ребра:** `from` влияет на `to` (снизу вверх по дереву влияния).

### 7.4. Раскладка графа (`layoutTree`)

1. Построить иерархию по **прямым** связям (`direct`).
2. Узлы без прямого родителя привязываются через первую **косвенную** связь.
3. Tidy-tree: родитель центрируется над детьми.
4. `manualPos` сохраняет координаты после drag-and-drop пользователя.
5. Кнопка «Вернуть авто-раскладку» сбрасывает `manualPos`.

### 7.5. Персистентность

**Ключ:** `metric-tree-builder-v2`

```json
{
  "tree": { ... },
  "lastInput": { "desc", "metrics", "text" },
  "manualPos": { "nodeId": { "x", "y" } },
  "showAllLinks": boolean,
  "ts": number
}
```

Автосохранение при изменениях дерева (`saveState()`).

### 7.6. Движок инсайтов (`computeInsights`)

Rule-based проверки (без ML):

| # | Проверка | Уровень |
|---|----------|---------|
| 1 | Наличие единственной apex-метрики L1 | warn/good |
| 2 | Драйверы L1–L2 без нижнего объяснения | warn |
| 3 | output/guardrail с direct-связью вверх | tip |
| 4 | Наличие guardrail-метрик | warn/good |
| 5 | Метрики без владельца | tip |
| 6 | Vanity по regex VANITY | warn |
| 7 | Изолированные узлы | tip |
| 8 | Пустые слои LEVELS | tip |
| 9 | Покрытие isCurrent + слепые зоны по слоям | good/tip |
| 9b | Валидация userAdded-метрик | warn/good |
| 10 | KEYWORDS из описания без узла в дереве | tip |

Группировка в UI: **Критично** / **Рекомендации** / **В порядке**.

### 7.7. Валидация узла (`validateNode`)

Правила для карточки «Разбор метрики»: связь вверх, тип связи, формула, владелец, vanity, расхождение слоя с библиотекой, дубликаты имён.

### 7.8. Библиотека метрик

- **~100+** предопределённых карточек в `LIBRARY`.
- **11 категорий** `CATEGORIES`: финансы, клиенты, продукт, сервис, драйверы, tech, payments, lending, neobank, wealth, AI/AIPDLC.
- Расширения: knowledge-продукты, AIPDLC-набор (HVR, AUE, IOD, CoL, DORA и др.), финтех-сегменты.
- Поиск: фильтр по name + what (клиентский, без полнотекстового индекса).

### 7.9. Экспорт

| Формат | Содержимое |
|--------|------------|
| JSON | Полный объект `tree` |
| SVG | Клон canvas + стили + тёмный фон |
| HTML | Standalone страница с SVG |
| PNG | Rasterization SVG через canvas (может блокироваться CORS в части браузеров) |

### 7.10. Встроенный self-test

При загрузке выполняется `selfTest()`: проверка `buildDemoTree`, `computeInsights`, `normalizeTree` на тестовом вводе. Ошибки → `console.error`.

## 8. Интеграции (текущие и планируемые)

### 8.1. Текущая интеграция с LLM

| Точка | Функция | Условие |
|-------|---------|---------|
| Построение дерева | `callAi` | `f_mode=api` + ключ |
| Консультант | `askAiConsult` | ключ задан |
| Разбор узла | `askAiAboutNode` | ключ задан |

Контракт: OpenAI Chat Completions, Bearer token, JSON в ответе.

### 8.2. Планируемая интеграция (не в прототипе)

- **GigaChat** / корпоративный LLM через server-side proxy.
- OAuth token flow на backend.
- Rate limiting, аудит запросов, маскирование чувствительных данных в логах.

## 9. UI/UX-требования (зафиксированные в прототипе)

- Тёмная тема, финтех-эстетика, анимированный фон (горизонтальные линии).
- Блок ввода **не скрывается** после генерации — прокрутка к дереву.
- Онбординг 3 шага + 2 примера продуктов.
- Подсказка «Как читать дерево» (структура, слои, рамки).
- Privacy note: локальный режим по умолчанию.
- Drawer для деталей метрики (не модальное окно на весь экран).

## 12. Критерии приёмки MVP (предложение для prod)

1. Пользователь описывает продукт → получает дерево 15–35 узлов за < 30 с (p95) через LLM.
2. Дерево проходит ≥ 8 из 10 rule-checks инсайтов без критичных warn.
3. Пользователь редактирует метрику и связи; изменения сохраняются в аккаунте.
4. Библиотека: поиск находит метрику по названию; добавление в дерево за 2 клика.
5. Экспорт JSON валиден по JSON Schema из §7.3.
6. Локальный/demo-режим доступен без сети (offline-first fallback).
